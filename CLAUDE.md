## GitHub ワークフロー（厳守）

> これらのルールは Claude Code としての動作に適用される強制ルールです。
> **いかなる例外も認めません。**

### 1. Issue ファーストの原則

- **作業を始める前に必ず GitHub Issue を作成すること**
- バグ修正・新機能・リファクタリング・ドキュメント更新のいかなる変更も Issue なしに着手してはならない
- Issue には以下を含めること:
  - 何をするのか（タイトル）
  - なぜするのか（本文）
  - 完了条件（できれば箇条書き）

### 2. ブランチ命名規則

**フォーマット:** `{type}/{短い説明}-#{番号}`

| type | 用途 |
|------|------|
| `feature` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメント変更 |
| `refactor` | リファクタリング（機能変更なし） |
| `test` | テスト追加・修正 |
| `chore` | 設定・依存関係などのメンテナンス |

**例:**
- `feature/add-task-labels-#12`
- `fix/task-drag-not-working-#34`
- `chore/setup-docker-#5`

**禁止パターン:**
- `main` への直接プッシュ
- `test-branch`, `tmp`, `wip` など説明のないブランチ名
- Issue 番号なしのブランチ

### 3. コミットメッセージ規則

**フォーマット:** `{type}: {変更内容の説明（日本語）}`

- **説明は日本語で記述すること**（既存の履歴に合わせる）
- type は英語のまま（`feat`, `fix` 等）

```
feat: タスクのラベル機能を追加

- ラベルのCRUD APIを実装
- フロントエンドのラベル選択UIを追加

Closes #12
```

| type | 用途 |
|------|------|
| `feat` | 新機能 |
| `fix` | バグ修正 |
| `docs` | ドキュメント |
| `refactor` | リファクタリング |
| `test` | テスト |
| `chore` | その他メンテナンス |
| `style` | フォーマット変更（動作変更なし） |

- 1行目は50文字以内を目安にする
- 関連 Issue は `Closes #番号` で紐付ける

### 4. プルリクエスト（PR）ルール

- **PR なしに main ブランチにマージしてはならない**
- PR には必ず対応する Issue を `Closes #番号` で参照すること
- PR タイトルはコミットメッセージと同じフォーマットに従う
- PR 本文には以下を記載する:
  1. 変更内容の概要
  2. テスト方法・確認手順
  3. 関連 Issue のリンク

### 5. 禁止事項

以下の操作は**絶対に行わない:**

- `git push origin main`（直接プッシュ）
- `git push --force` / `git push -f`（強制プッシュ）
- Issue を作成せずにブランチを切る
- PR なしで main にマージする
- 命名規則に違反したブランチ名を使う
